# 향후 데이터 소스 후보 (Future Weather Sources)

현재 MVP에는 **포함하지 않았지만** 가치가 있어 기록해 두는 후보들입니다. 의도적으로 **과설계하지 않았습니다** — 각 항목은 "무엇을 더 줄 수 있나 / 왜 지금은 아닌가 / 들어온다면 어떻게"만 간단히 적습니다. 현재 구현된 소스는 [`weather-sources.md`](weather-sources.md) 참조.

도입 시 공통 원칙(현재와 동일): **서버 사이드 전용**, 키 0개 기반은 유지, 미설정·실패 시 조용히 강등, 데이터 위조 금지, 표시 시 출처/약관 준수.

---

## 1. Meteostat — 과거 관측·기후 평년값

- **줄 수 있는 것**: 장기 과거 관측과 평년값(climate normals). "예년보다 X도 높음" 같은 **맥락 카피**, 또는 시네마틱 무드의 계절 보정에 쓸 수 있음.
- **지금은 아닌 이유**: SeoulSky는 *지금 이 순간*의 하늘을 그리는 실시간 경험. 과거/평년값은 핵심 루프에 불필요. 추가 호출·캐시 복잡도 대비 시각적 이득이 작음.
- **들어온다면**: 별도 `lib/providers/meteostat.ts` + 일 단위 캐시(평년값은 거의 불변). 카피 레이어에서만 소비, 씬 물리에는 미반영. 무료 티어/약관 확인 필요.

## 2. ECMWF (예: Open-Meteo의 ECMWF 모델, 또는 ECMWF 오픈 데이터)

- **줄 수 있는 것**: 세계 최고 수준의 결정론적/앙상블 모델. 앙상블 스프레드는 **예보 신뢰도**를 모델 자체에서 산출하는 데 이상적.
- **지금은 아닌 이유**: 현재 신뢰도는 Open-Meteo↔MET Norway **소스 간 일치도**로 충분히 표현됨. 원시 앙상블은 데이터량·해석 비용이 큼. ECMWF 오픈 데이터는 **GRIB2** 배포가 많아 부담(아래 경고 참조).
- **들어온다면**: 원시 GRIB 대신 **Open-Meteo가 제공하는 ECMWF 모델 파라미터**(`models=ecmwf_ifs025` 등)를 JSON으로 받는 경로가 현실적. 비교 엔진에 제3 소스로 추가.

## 3. NOAA GFS — 미국 글로벌 모델

- **줄 수 있는 것**: 완전 공개 도메인 글로벌 모델. 다른 모델과의 **모델 간 비교**로 신뢰도 다양성 확보.
- **지금은 아닌 이유**: 원천 배포가 GRIB2(NOMADS/AWS Open Data). 한 지점(서울)만 필요한데 글로벌 격자를 다루는 건 과함.
- **들어온다면**: 마찬가지로 Open-Meteo의 GFS 모델(`models=gfs_global`) JSON 경로 권장. 직접 GRIB 파이프라인은 비권장.

## 4. DWD ICON — 독일 기상청 모델

- **줄 수 있는 것**: 고해상도 글로벌/유럽 모델. 단기 예보 품질이 우수해 비교군으로 좋음.
- **지금은 아닌 이유**: NOAA/ECMWF와 동일 — 원시 배포가 GRIB2, 단일 지점에 과한 인프라.
- **들어온다면**: Open-Meteo의 ICON 모델(`models=icon_seamless`) JSON.

---

## ⚠️ GRIB을 브라우저/엣지에서 직접 파싱하지 말 것

ECMWF·GFS·ICON의 **원시 배포 형식은 GRIB2**입니다. 다음 이유로 본 프로젝트의 런타임(Next.js, 브라우저/엣지)에서 직접 디코딩하는 것은 **권장하지 않습니다**:

- **무겁다**: 단일 GRIB 메시지도 글로벌 격자라 수 MB~수십 MB. 서울 한 점을 위해 받기엔 과도.
- **복잡하다**: GRIB2는 템플릿·비트맵·복합 패킹(JPEG2000/PNG/복소수)으로 정확한 디코딩이 까다롭고, 순수 JS 파서는 부분적이거나 느림.
- **잘못된 위치**: 디코딩·서브셋·단위 변환은 서버/배치 작업이지 요청 경로의 일이 아님. 브라우저에서 하면 번들 비대·메인 스레드 블로킹·메모리 급증.

**권장 경로**: 이 모델들이 필요하면 GRIB을 직접 다루지 말고, 이미 모델별 파라미터를 **JSON으로 정규화해 주는 Open-Meteo의 `models=` 파라미터**를 통해 받습니다(현재 Open-Meteo 프로바이더 패턴 그대로 확장). 정말로 원시 모델이 필요하면 별도 **백엔드 사전처리 서비스**에서 서울 지점만 추출해 JSON으로 캐싱한 뒤 본 앱이 그 JSON만 소비해야 합니다.

## 도입 우선순위 메모

추가한다면 가치/비용 순으로: (1) **ECMWF via Open-Meteo** — 비교 신뢰도 다양성, 거의 무비용. (2) **GFS/ICON via Open-Meteo** — 모델 간 비교 강화. (3) **Meteostat** — 카피용 맥락, 선택적. 원시 GRIB 파이프라인은 실시간 단일 지점 경험에는 정당화되지 않음.
